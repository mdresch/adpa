# Implementation Plan: IP & Research Novelty Governance Pipeline

**Date**: 2026-06-15  
**Priority**: HIGH — Strategic differentiation and IP asset protection  
**Spec ID**: RPAS-CM-ENV-IP-001  
**Governed Feature IDs to Create**: `ip-novelty-assessment`, `ip-legal-review-workflow`, `ip-registration-pipeline`

---

## Goal Description

Positive drift detection in ADPA already fires a `driftCategory: 'innovation'` classification inside `positiveDriftChangeRequestService.ts` (line 105) with the note *"Potential intellectual property or competitive differentiation"* — but the pipeline **stops there**. There is no automated mechanism to:

1. **Rigorously investigate** whether the detected artifact constitutes genuine Intellectual Property or Research Novelty originating from the ADPA system.
2. **Route to a legal team** for formal review.
3. **Approve or reject** the novelty claim through a governance workflow.
4. **Register** approved IP claims at the appropriate agencies (USPTO, EPO, WIPO, Copyright Office, etc.).

This plan wires the existing `innovation` drift hook directly into a full **IP Governance & Registration Pipeline** — a completely new governance pillar built on top of the proven `approvalWorkflowService` and `escalationService` infrastructure.

---

## Architecture: The IP Pipeline

```
[Drift Detection]
       │
       │ driftCategory === 'innovation'
       │ OR drift.description includes 'novel', 'patent', 'unique method'
       ▼
[IPNoveltyAssessmentService]          ← NEW
  ├── Automated Prior Art Search      (queries public patent APIs: USPTO, EPO, Google Patents)
  ├── ADPA System Origin Verification (confirms artifact was ADPA-generated, not imported)
  ├── Novelty Score Calculation       (0.0 – 1.0 confidence of genuine novelty)
  └── IP Classification               (Patent / Copyright / Trade Secret / None)
       │
       │ noveltyScore >= 0.60
       ▼
[IPLegalReviewWorkflow]               ← NEW (extends approvalWorkflowService)
  ├── Creates 'ip_novelty_review' approval request
  ├── Routes to Legal Team role
  ├── Attaches: Prior Art Report, System Origin Evidence, Novelty Assessment
  ├── Legal team can: APPROVE / REJECT / REQUEST_INVESTIGATION
  └── SLA: 10 business days (configurable)
       │
       │ status === 'APPROVED'
       ▼
[IPRegistrationPipeline]              ← NEW
  ├── Formats claim for target agency (USPTO, EPO, WIPO, Copyright Office)
  ├── Generates draft filing document (stored as ADPA document)
  ├── Tracks filing status (DRAFT → FILED → PENDING → GRANTED / REJECTED)
  └── Notifies stakeholders on status changes
```

## Feature 1: IP Novelty Assessment Service

### Hook Point
Modify `positiveDriftChangeRequestService.ts` `analyzePositiveDrift()` at the existing `driftCategory = 'innovation'` branch (lines 100–108). Instead of estimating a fixed `innovationValue += 50000`, trigger the new `IPNoveltyAssessmentService`.

### `[NEW] server/src/modules/ip-governance/IPNoveltyAssessmentService.ts`

```typescript
interface NoveltyAssessment {
  noveltyScore: number                        // 0.0–1.0
  ipClassification: 'patent' | 'copyright' | 'trade_secret' | 'none'
  priorArtFound: PriorArtResult[]             // USPTO/EPO search results
  systemOriginVerified: boolean               // Confirmed ADPA-generated
  evidenceDocumentId: string                  // Pointer to evidence document
  recommendedAction: 'file_patent' | 'file_copyright' | 'mark_trade_secret' | 'no_action'
  estimatedIPValue: number                    // Monetized estimate ($ over 10 years)
}
```

**Methods**:
- `assessNovelty(driftRecord, driftPoints, entityProvenance)` — master orchestration method.
- `searchPriorArt(description, technicalDomain)` — queries patent databases via public APIs (USPTO Open Data Portal, EPO's OPS API, Google Patents via SerpAPI or similar).
- `verifySystemOrigin(entityId, sourceDocumentId)` — uses the Entity Provenance Chain (from the Dependency Graph plan) to confirm the artifact was generated by ADPA, not imported or plagiarized.
- `calculateNoveltyScore(priorArtResults, systemOriginVerified, driftDescription)` — weighted scoring function.
- `generateEvidenceDocument(assessment, driftRecord)` — produces a structured ADPA document containing all evidence for the legal team.

### Prior Art Search Integration

External APIs to integrate (all public/free tiers available):
- **USPTO Open Data Portal**: `https://developer.uspto.gov/` — US patent full-text search.
- **EPO OPS API**: `https://ops.epo.org/` — European and PCT patent search.
- **Google Patents SerpAPI**: Structured patent metadata search.

The search query is constructed from the drift description and the extracted entity labels to find semantically similar existing patents.

## Feature 2: IP Legal Review Workflow

### `[NEW] server/src/modules/ip-governance/IPLegalReviewWorkflow.ts`

Extends the existing `approvalWorkflowService` with a new `request_type: 'ip_novelty_review'`.

**Approval Request Payload**:
```typescript
interface IPLegalReviewRequest {
  request_type: 'ip_novelty_review'
  ip_assessment_id: string
  drift_record_id: string
  project_id: string
  title: string                               // e.g. "IP Novelty Review: Novel RAG Context Injection Method"
  novelty_score: number
  ip_classification: string
  prior_art_summary: string
  system_origin_evidence: string
  evidence_document_id: string               // ADPA document with full evidence package
  estimated_ip_value: number
  recommended_action: string
  priority: 'low' | 'medium' | 'high'
  sla_business_days: number                  // Default: 10
}
```

**Legal Team Actions**:
- `APPROVE_FOR_FILING` — escalates to `IPRegistrationPipeline` for the recommended registration type.
- `APPROVE_AS_TRADE_SECRET` — marks as internal Trade Secret, restricts access in ADPA.
- `REQUEST_DEEPER_INVESTIGATION` — re-opens `IPNoveltyAssessmentService` with additional context.
- `REJECT` — records reason, closes the IP claim. Entity is tagged `ip_reviewed_negative`.

**Notifications**:
- Legal team receives an email via `emailNotificationService` with a pre-formatted **IP Novelty Report** as attachment (generated by ADPA's document generation engine).
- Uses the existing Teams channel notification for real-time alert.

## Feature 3: IP Registration Pipeline

### `[NEW] server/src/modules/ip-governance/IPRegistrationPipeline.ts`

Responsible for formatting and tracking IP filings after legal team approval.

**Supported Agencies & Formats**:
| Agency | Filing Type | Format | Notes |
|--------|------------|--------|-------|
| USPTO | Utility Patent | Provisional Application (PPA) format | AI-generated draft via ADPA document engine |
| EPO | European Patent | EP-A format | Requires PCT or direct national route |
| WIPO | PCT International | PCT Request (PCT/RO/101) | Broadest international protection |
| Copyright Office | Software Copyright | Form TX (literary works) | Most applicable to AI-generated text artifacts |
| Internal | Trade Secret | ADPA Trade Secret Registry | Encrypted, access-controlled document |

**Methods**:
- `initiateRegistration(approvedIPClaim, targetAgency)` — creates the draft registration document using ADPA's template engine.
- `updateFilingStatus(ipClaimId, newStatus, filingReference)` — tracks USPTO/EPO reference numbers.
- `scheduleMaintenanceReminders(ipClaimId, filingType)` — schedules renewal reminders (e.g., patent maintenance fees at 3.5, 7.5, 11.5 years for USPTO utility patents).

**Filing Status Machine**:
```
DRAFT → LEGAL_APPROVED → FILED → PENDING_EXAMINATION → 
  GRANTED (→ MAINTENANCE_SCHEDULED) | REJECTED (→ APPEAL or CLOSED)
```

### `[NEW] Database Schema`
```sql
CREATE TABLE ip_claims (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  project_id UUID NOT NULL REFERENCES projects(id),
  drift_record_id UUID REFERENCES baseline_drift_detection(id),
  source_entity_ids UUID[],                          -- Entities that constituted the novelty
  source_document_ids UUID[],                        -- Provenance documents
  novelty_score DECIMAL(3,2),
  ip_classification TEXT,                            -- patent | copyright | trade_secret
  prior_art_found BOOLEAN DEFAULT FALSE,
  system_origin_verified BOOLEAN DEFAULT FALSE,
  legal_review_status TEXT DEFAULT 'pending',        -- pending | approved | rejected | investigating
  legal_reviewer_id UUID,
  filing_status TEXT DEFAULT 'draft',               -- draft | filed | pending | granted | rejected
  target_agency TEXT,                                -- USPTO | EPO | WIPO | COPYRIGHT | INTERNAL
  agency_reference_number TEXT,                      -- e.g., '16/123,456' (USPTO Application No.)
  estimated_ip_value DECIMAL(15,2),
  evidence_document_id UUID REFERENCES documents(id),
  filing_document_id UUID REFERENCES documents(id),
  filed_at TIMESTAMPTZ,
  granted_at TIMESTAMPTZ,
  next_maintenance_due TIMESTAMPTZ,
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE ip_claim_entities (
  ip_claim_id UUID REFERENCES ip_claims(id),
  entity_id UUID NOT NULL,
  entity_role TEXT NOT NULL,  -- 'primary_invention' | 'supporting_evidence' | 'prior_art_reference'
  PRIMARY KEY (ip_claim_id, entity_id)
);
```

## Verification Plan

### Automated Tests (Contract Guards)

| ID | Invariant |
|----|-----------|
| `REQ-IP-001` | `assessNovelty()` MUST return `noveltyScore = 0` if `systemOriginVerified = false` — unverified origin cannot be claimed as ADPA IP |
| `REQ-IP-002` | `assessNovelty()` MUST return `ipClassification = 'none'` if prior art is found with similarity > 0.85 |
| `REQ-IP-003` | An IP claim CANNOT be created without a linked `evidence_document_id` — all claims require an evidence package |
| `REQ-IP-004` | Any IP claim with `estimatedIPValue > $500,000` MUST pass the DRACO gate before routing to legal review |
| `REQ-IP-005` | `IPRegistrationPipeline.initiateRegistration()` MUST NOT be callable unless `legal_review_status = 'approved'` |
| `REQ-IP-006` | Filing status MUST be logged to `audit_logs` at every state transition |

### Manual Verification
1. Introduce a drift point with `description: "Novel automated context injection method using temporal entity graphs"`.
2. Verify `analyzePositiveDrift()` classifies as `driftCategory = 'innovation'` and triggers `IPNoveltyAssessmentService`.
3. Verify prior art search runs and returns results (even empty is valid).
4. Verify `system_origin_verified = true` via entity provenance lookup.
5. Verify a legal review approval request appears in the approval dashboard for the Legal team role.
6. Approve the claim as Legal reviewer.
7. Verify `IPRegistrationPipeline` generates a draft Patent Application document accessible in the project's document library.
8. Verify the IP Claims Registry on the frontend shows the claim with status `DRAFT`.

## Open Questions

> [!IMPORTANT]
> **Legal Roles Configuration**: Who in the system is assigned the "Legal Team" role for IP review routing? Is there an existing role hierarchy, or does this require a new `legal_reviewer` role type to be added to the user management system?

> [!IMPORTANT]
> **Patent API Access**: USPTO OPS and EPO OPS are free but require API key registration. Should these be added to `server/.env` as optional integrations (graceful fallback to manual search if keys are absent)?

> [!NOTE]
> **Trade Secret vs Patent Choice**: For AI-generated methods, Trade Secret protection (keeping the algorithm private) is often stronger than a Patent (which requires public disclosure). The legal review workflow should prompt the Legal team to explicitly choose the protection strategy, not just approve filing.
